# DEPLOYMENT.md — איך VaddyGo עולה לאוויר

## התמונה הגדולה

```
את דוחפת קוד ל-main  →  GitHub Actions בודק (טסטים + build)  →  Railway פורס אוטומטית  →  האתר מתעדכן
```

אם הבדיקות נכשלות — Railway לא פורס, והגרסה הקודמת ממשיכה לרוץ. ככה אי אפשר "לשבור" את האתר בטעות.

## CI — GitHub Actions

הקובץ [.github/workflows/ci.yml](.github/workflows/ci.yml) רץ אוטומטית על כל push ל-main. הוא: מתקין תלויות → מריץ טסטים → בונה את האפליקציה. רואים את התוצאה בטאב **Actions** בעמוד הריפו ב-GitHub (✅ ירוק = עבר, ❌ אדום = נכשל).

## Railway — הקמה ראשונית (עושים פעם אחת)

1. נכנסים ל-https://railway.app ולוחצים **Login** → **Login with GitHub** (מתחברים עם אותו חשבון GitHub).
2. לוחצים **New Project** → **Deploy from GitHub repo**.
3. בפעם הראשונה Railway יבקש הרשאה לגשת ל-GitHub — לוחצים **Configure GitHub App**, בוחרים את הריפו `avivit-viskin/vaddygo` ומאשרים.
4. בוחרים את `vaddygo` מהרשימה. Railway יתחיל Build אוטומטית.
5. **Start Command:** נכנסים לשירות שנוצר → **Settings** → **Deploy** → בשדה **Custom Start Command** כותבים:
   ```
   npx serve -s build -l 3000
   ```
   (זה אומר ל-Railway: אחרי הבנייה, הגש את התיקייה `build` כאתר סטטי על פורט 3000.)
6. **Wait for CI:** באותו עמוד Settings, מחפשים את המתג **Wait for CI** ומדליקים אותו. זה מה שגורם ל-Railway לחכות שהבדיקות ב-GitHub יעברו לפני פריסה.

## איך מקבלים דומיין (כתובת לאתר)

בהתחלה לשירות ב-Railway **אין כתובת ציבורית** — צריך לבקש אחת (זה חינם):

1. בעמוד הפרויקט ב-Railway לוחצים על השירות (הקופסה עם שם הריפו).
2. עוברים לטאב **Settings**.
3. גוללים לחלק **Networking** (או **Public Networking**).
4. לוחצים על **Generate Domain**. אם נשאלת על פורט — מזינים **3000** (חייב להתאים לפורט שב-Start Command).
5. Railway ייצור כתובת בסגנון `vaddygo-production.up.railway.app` — זו הכתובת של האתר! לוחצים עליה כדי לפתוח.

בהמשך, אם תרצי דומיין משלך (למשל `vaadygo.co.il`), קונים דומיין ובאותו מקום לוחצים **Custom Domain** — אבל בשלב הזה הדומיין החינמי מספיק לגמרי.

## פריסת הבקאנד (השרת) ל-Railway — מדריך לחיצה-לחיצה

הקוד כבר מוכן (Dockerfile + האזנה לפורט). כל מה שנשאר הוא בדשבורד של Railway. מונחים: **קנבס** = המסך האפור הגדול בתוך הפרויקט שעליו קופסאות; כל **קופסה** = שירות אחד. **Volume** = דיסק קבוע ששומר את הנתונים גם כשמעדכנים גרסה.

### חלק א — יוצרים את שירות השרת

1. נכנסים ל-https://railway.app ולוחצים על הפרויקט שלך. עכשיו רואים את הקנבס עם קופסה אחת (האתר).
2. לוחצים על כפתור **+ Create** בפינה הימנית-העליונה של המסך (אם אין — קליק ימני על שטח ריק בקנבס).
3. נפתח תפריט. בוחרים **GitHub Repo** → ברשימה לוחצים על **avivit-viskin/vaddygo**.
4. תופיע קופסה חדשה על הקנבס והיא תתחיל לבנות. ייתכן שהבנייה הראשונה תיכשל (❌ אדום) — זה תקין, עוד לא הגדרנו אותה.

### חלק ב — מגדירים Root Directory (איפה השרת בריפו)

5. לוחצים קליק אחד על הקופסה החדשה. נפתח חלון גדול (בצד או במרכז).
6. בחלק העליון של החלון יש טאבים: **Deployments · Variables · Metrics · Settings**. לוחצים **Settings**.
7. גוללים עד הכותרת **Source**. שם יש שורה **Root Directory** עם אייקון עיפרון/כפתור Add. לוחצים, כותבים `backend`, מאשרים (Enter או ✓).
8. באותו עמוד Settings מדליקים גם את המתג **Wait for CI** (אם מופיע).

### חלק ג — מחברים דיסק קבוע (Volume)

9. סוגרים את החלון (X למעלה או לחיצה על הרקע האפור) — חוזרים לקנבס.
10. לוחצים **קליק ימני** על קופסת השרת. נפתח תפריט → בוחרים **Attach Volume**.
11. נפתח חלון קטן עם שדה **Mount Path**. כותבים בדיוק: `/data` ולוחצים **Attach**.
12. תופיע קופסה קטנה של דיסק מחוברת בקו לקופסת השרת. זהו — הנתונים ישרדו עדכונים.

### חלק ד — משתני סביבה לשרת

13. לוחצים על קופסת השרת → טאב **Variables** (ליד Settings).
14. לוחצים **+ New Variable**. מופיעים שני שדות: שם (VARIABLE_NAME) וערך (VALUE).
15. משתנה ראשון — שם: `ConnectionStrings__Default` (שימי לב: **שני** קווים תחתונים רצופים __), ערך: `Data Source=/data/vaadygo.db` → לוחצים **Add**.
16. שוב **+ New Variable** — שם: `Cors__AllowedOrigins__0`, ערך: הכתובת של **האתר** שלך (למשל `https://vaddygo-production.up.railway.app`) — בלי `/` בסוף. הכתובת כתובה על קופסת האתר בקנבס. → **Add**.
17. למעלה יופיע פס סגול עם כפתור **Deploy** — לוחצים עליו כדי שהשינויים ייכנסו לתוקף.

### חלק ה — דומיין לשרת

18. עדיין בקופסת השרת: טאב **Settings** → גוללים ל-**Networking** (או Public Networking) → **Generate Domain**.
19. אם נשאלת על פורט — כותבים **8080**.
20. מופיעה כתובת חדשה (למשל `vaddygo-backend-production.up.railway.app`). **מעתיקים אותה** — נצטרך אותה בצעד הבא.

### חלק ו — מחברים את האתר לשרת

21. סוגרים את החלון וחוזרים לקנבס. לוחצים על קופסת **האתר** (הראשונה).
22. טאב **Variables** → **+ New Variable** — שם: `REACT_APP_API_URL`, ערך: כתובת השרת מצעד 20, עם `https://` בהתחלה ובלי `/` בסוף, **בלי** להוסיף `/api`. → **Add** → **Deploy** בפס הסגול.

### בדיקה — איך יודעים שהצליח

23. מחכים שבטאב **Deployments** (בשני השירותים) יופיע ✅ ירוק.
24. גולשים בדפדפן אל: כתובת-השרת + `/api/students` (למשל `https://vaddygo-backend-production.up.railway.app/api/students`). אמור להופיע: `[]`.
25. פותחים את האתר עצמו → מסך התלמידים נטען בלי הודעת שגיאה, וכל תלמידה שמוסיפים נשמרת באמת.

**טעויות נפוצות:** קו תחתון אחד במקום שניים בשמות המשתנים · `/` מיותר בסוף כתובת · שכחת ללחוץ Deploy אחרי הוספת משתנים.

## דברים חשובים לדעת
- **משתני סביבה:** בטאב **Variables** של השירות ב-Railway מגדירים משתנים כמו `REACT_APP_API_URL` (יוגדר בשלב 1). לעולם לא שומרים סודות בקוד.
- **לוגים:** טאב **Deployments** → לוחצים על הפריסה → **View Logs**. שם רואים למה build נכשל אם נכשל.
- **Rollback (חזרה אחורה):** בטאב **Deployments** רואים את כל הפריסות הקודמות. ליד פריסה ישנה שעבדה לוחצים על ⋮ → **Redeploy** — והאתר חוזר לגרסה ההיא תוך דקות.

## בעיות נפוצות

| בעיה | פתרון |
|------|--------|
| Build נכשל ב-Railway | לבדוק ב-View Logs; לרוב חסר Start Command או שה-build נכשל גם מקומית (`npm run build`) |
| האתר מציג דף ריק | לוודא שה-Start Command הוא בדיוק `npx serve -s build -l 3000` ושפורט הדומיין הוא 3000 |
| הפריסה לא קורית אחרי push | לבדוק שה-CI ירוק בטאב Actions ב-GitHub; אם אדום — Railway מחכה בכוונה |
